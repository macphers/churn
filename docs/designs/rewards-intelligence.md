## Problem
The app treats all rewards points as interchangeable. 85,000 Chase UR (worth ~$1,700) looks the same as 85,000 Delta SkyMiles (~$1,020) or 85,000 Hilton points (~$425). This makes the app a toy, not a tool.

## Solution
Program-aware data model with built-in program database, point valuations, transfer partners, earn rates, and portfolio valuation in dollars.

## Architecture

### Two-Level Data Model: Program → Cards
One account = one rewards program with one shared balance. Cards are sub-entities with earn rates and annual fees.

Example: Chase Ultimate Rewards (85,000 pts) contains Sapphire Preferred (3x dining, $95/yr) and Freedom Unlimited (1.5x everything, $0/yr).

### Built-in Program Database (~15-20 programs)
Inline JS constant with known programs, cpp values, transfer partners, and card earn rates. User selects from searchable dropdown; values auto-populate. Custom programs supported for anything not in the database.

### Schema v3
```
account: {
  id, programSlug, program, type, balance, currency,
  cpp,              // cents per point (from DB, user-editable)
  accountNumber, lastUpdated, notes,
  cards: [{
    name,           // "Sapphire Preferred"
    annualFee,      // 95
    earnRates: {    // multipliers per category
      dining, travel, groceries, gas, streaming, onlineShopping, other
    }
  }]
}
```

Migration v2→v3: existing accounts get `programSlug: null`, `cpp: 1.0`, `cards: []`.

### Dashboard
- Hero shows **dollar value** as primary number ("$3,145") with points as subtitle
- Points Distribution chart shows dollar values
- Inline dollar value on every card ("85,000 pts ≈ $1,700")

### Add Account Form
Progressive disclosure in one modal:
1. Program picker (searchable dropdown + "Custom")
2. Auto-filled fields (cpp, currency, type)
3. Balance + account number
4. Cards sub-section (add cards with earn rates + annual fee)
5. Notes

### Earn Rate Categories (7)
Dining, Travel, Groceries, Gas, Streaming, Online Shopping, Other
